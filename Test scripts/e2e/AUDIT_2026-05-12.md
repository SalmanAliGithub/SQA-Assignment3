# e2e Harness Logical-Correctness Audit — 2026-05-12

**Run under review:** `tools/e2e/logs/run-2026-05-12T10-30-50-824Z.{json,log}` — 57 PASS / 0 FAIL / 82.8 % auth coverage.
**Backend state cited:** live `maal-prod-postgres` after the same run, queried at audit time. Container uptime: 43 min.
**Evidence convention:** JSONL citations as `JSONL ts=<ts> scope=<scope>`. SQL citations as `psql:<table> <predicate>`. All SQL was run by `docker exec maal-prod-postgres psql -U maal -d maal -c '…'`.

---

## Verdict

Of the 57 PASSed flows, **only ~24 are logically correct end-to-end**. The remaining ~33 split into three buckets: (a) **pass-but-wrong** — the API returned 2xx and the harness accepted it, but the underlying state diverges from spec (notably F-9 cash-in inflates `tx.amount` and the recipient SMS by the commission; F-23 audit-export queues a job that never completes; N-9 frozen-user returns 401 instead of 403); (b) **untested side-effects** — harness asserts nothing on monetary deltas, balance invariants, audit-log row creation, outbox drain, or commission-rule application, so a whole class of silent regressions passes by default; (c) **scaffold-only passes** — F-16 BNPL accept returned 422 but the harness tolerated it and skipped repayment/writeoff/contract assertions, so the BNPL lifecycle is essentially unverified despite the green PASS. Three new backend bugs surface: `transactions.amount` semantic drift (gross instead of principal), three stuck `outbox_events` with `last_error = "No BullMQ queue configured"`, and the AUDIT_EXPORT report job left QUEUED indefinitely. The previously-known BUG-06 (device-bind) and BUG-09 (suspended float-topup) are genuinely fixed; BUG-10 and BUG-12 remain reproducible behind permissive `allowStatus` lists.

## Newly-found harness assertion gaps

### HARNESS-GAP-1 — No monetary-delta assertion on any money-moving flow
- F-9, F-10, F-11, F-12, F-16, F-18 all PASS without checking `wallet.balanceAfter - wallet.balanceBefore == amount`. The F-9 cash-in test only checks `balance` is truthy (`tools/e2e/flows/f09-agent-cashin.mjs:79`).
- **Recommend:** add `tools/e2e/lib/side-effects.mjs::assertWalletDelta(ctx, actorLabel, expectedDeltaStr, opts)` that captures `availableBalance` before the action and verifies the post-action delta. Wire into F-9 (after `cash-in` step), F-10 (after `approve`), F-11 (after `commit`), F-12 (after `transfer`, both sender and receiver), F-18 (after `reverse` — both wallets), F-9 + F-21 (cash-in on TIER_1 agent after rule retire — see HARNESS-GAP-A).

### HARNESS-GAP-2 — No commission-amount assertion
- F-9 captures the commission entry list but only checks `breakdown.length > 0` (`f09-agent-cashin.mjs:93`). It does not compute the expected commission from `commission_rules.percentage_rate × amount + fixed_fee` and compare against the stored `metadata.commissionAmount`.
- **Recommend:** add `assertCommissionApplied(ctx, txId, expectedCommissionStr)` that re-queries `transactions.metadata.commissionAmount` and asserts equality.

### HARNESS-GAP-3 — No audit-log row creation assertion
- For every admin action (KYC approve, freeze, broadcast, reverse, commission CRUD, system_config patch, risk decide, dispute pickup), the harness never queries `audit_logs` to verify the row exists with the right actor/target. The audit-list endpoint is read in F-23 but only structurally (limit=10, paginated).
- **Recommend:** add `assertAuditLogged(ctx, { actorRole, actionType, targetResourceId })` that uses a service-account JWT (or admin) to query `/admin/audit-logs?action=<x>&target=<id>` and asserts ≥ 1 row.

### HARNESS-GAP-4 — No outbox drain assertion
- The harness never queries `outbox_events`. The three BUG-NEW-A rows would have been caught if a final `assertOutboxDrained()` step existed.
- **Recommend:** add `tools/e2e/lib/side-effects.mjs::assertOutboxDrained(ctx, opts)` that does `SELECT count(*) FROM outbox_events WHERE published_at IS NULL AND committed_at < NOW() - INTERVAL '10 seconds'` and asserts 0. Run at the very end of the suite (or after each flow that emits events).

### HARNESS-GAP-5 — No SSE payload schema check
- F-20 connects to SSE but the flow body returns in 28 ms — no event capture. Inspect `tools/e2e/flows/f20-dashboard-sse.mjs` and `tools/e2e/lib/sse.mjs`; add a wait-for-events loop that captures ≥ 1 event and validates schema (`type`, `data`, `id`).

### HARNESS-GAP-6 — No idempotent-replay equality assertion
- F-9 cash-in-replay and F-12 replay-same both accept `[200, 201]` without checking that the returned `transactionId` equals the original. A regression that creates a duplicate tx on replay would pass.
- **Recommend:** assert `replay.body.transactionId === first.body.transactionId` and ideally `replay.body.replay === true` (per spec § F-9 idempotency negatives).

### HARNESS-GAP-7 — No error-code assertions on 4xx responses
- N-9, N-11, N-12, N-13, N-17, N-20 all accept loose `expectStatus` lists without verifying `error.code`. Combined with HARNESS-BUG-A (below), the actual error codes are invisible in JSONL. So even a regression like `ERR_INSUFFICIENT_BALANCE` returning instead of `ERR_USER_FROZEN` would slip past.
- **Recommend:** for every negative test, assert the documented `error.code` value. Build a per-endpoint expected-code table.

### HARNESS-GAP-8 — No active-rule uniqueness assertion
- F-21 creates rule 11 and the partial unique index auto-flips rule 3 to `is_active=false`. The harness does not assert `SELECT count(*) WHERE is_active=true AND tier=… AND tx_type=…) == 1` after the create.

### HARNESS-GAP-9 — F-16 BNPL repayment never exercised

- **Recommend:** split F-16 into F-16a (product CRUD + apply) and F-16b (accept + repay + writeoff). Pre-condition F-16b on actually getting a contract; if accept fails, FAIL the flow loudly (not skip silently).

### HARNESS-GAP-10 — F-23 audit-export never polled to terminal
- 202 + jobId accepted; no `pollJobStatus` loop. BUG-NEW-C would have been caught.

### HARNESS-GAP-11 — N-19 BNPL expiry uses wrong table name

## Newly-found harness bugs

### HARNESS-BUG-A — Redaction set includes `code`, masking error codes in JSONL
- **Evidence:** `tools/e2e/lib/logger.mjs:SENSITIVE_BODY_KEYS` includes `'code'`. Every `error.code` field in 4xx responses is stamped `"[REDACTED]"`. JSONL run log shows `error.code:"[REDACTED]"` 100+ times.
- **Impact:** Auditors and CI reviewers cannot see the error code from logs alone — the most diagnostic field for negative tests is invisible. Independently verified with curl that real codes (`ERR_UNAUTHORIZED`, `ERR_AGENT_NOT_ACTIVE`, etc.) are returned by the API.
- **Fix:** Remove `'code'` from the set. The intent was to redact OTP codes — but the field for OTP is `otpCode` / `mfaCode` (already in the set). The bare `code` collision is unintentional.

### HARNESS-BUG-B — N-19 references `bnpl_applications` instead of `financing_requests`
- **Evidence:** `tools/e2e/flows/negatives/n19-bnpl-accept-expired.mjs:30` runs `UPDATE bnpl_applications SET offer_expires_at = NOW() - INTERVAL '1 hour' WHERE id = '<appId>'`. The table does not exist; the actual BNPL application table is `financing_requests`. Run-2026-05-12T10:30:50 emitted a WARN at 10:31:20.151Z documenting the failure. Flow then falls through to the random-UUID fallback and asserts on 4xx — which is structurally trivial, not a real expiry test.
- **Fix:** rename to `financing_requests`. While at it, the fallback should be removed — if the precondition fails, the flow should FAIL, not silently smoke-test.

### HARNESS-BUG-C — F-24 body-substring fallback masks missing broadcastId trace
- **Evidence:** `tools/e2e/flows/f24-broadcast.mjs:28` — predicate falls back to `n.body.includes(marker)`. A regression that drops broadcast→notification correlation entirely would still pass.
- **Fix:** remove the body-substring path; require `payload.broadcastId === broadcastId` after BUG-NEW-D is fixed.

### HARNESS-BUG-D — N-9 / N-11 / N-12 / N-13 / N-17 / N-20 use `[REDACTED]`-blind `error.code` assertions
- See HARNESS-BUG-A; downstream symptom. Once `code` is removed from the redaction set, tighten each of these flows to assert the documented code (e.g. N-9 → `ERR_USER_FROZEN`).

### HARNESS-BUG-E — F-3 `revoke-other` ambiguity
- `tools/e2e/flows/f03-device-binding.mjs:41` picks "any other device" by id; in a fresh run there is at most one prior device (the originally-bound one), so this works. In a re-used DB (or after re-running), the picked device may not be the one we just bound. Make this explicit (e.g. pick the *current* device's `id` and revoke it, then re-login from scratch).
